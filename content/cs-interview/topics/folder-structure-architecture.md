# Folder Structure & Architecture

## 한 줄 정의

코드를 **어디에 두느냐**가 팀의 **경계(도메인·기능·계층)** 와 확장 비용을 결정한다.

## 카드 요약

- **기능별(feature-based)**: `features/auth`, `features/cart` — 도메인 응집, 변경 시 한 폴더 안에서 끝.
- **계층별(layered)**: `components`, `hooks`, `services` — 작은 규모에 단순, 커지면 기능이 흩어짐.
- **혼합**: `app` 라우트(Next/Expo Router) + `features` 하위에 UI·로직·API 묶기.
- **아키텍처 키워드**: **관심사 분리**, **의존성 방향**(UI → domain → infra), **공유 커널** 최소화.

## 면접 포인트

1. **왜 feature 폴더인가?** — 삭제·이동 시 영향 범위가 명확.
2. **공유 컴포넌트**: `ui/` 또는 디자인 시스템 레벨로만 올리고, 남용하면 진흙탕.
3. **테스트·스토리북** 위치도 구조와 같이 가야 유지보수 쉬움.
