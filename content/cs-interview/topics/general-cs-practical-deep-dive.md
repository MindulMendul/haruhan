# General CS Practical Deep Dive

## 한 줄 정의

일반 CS 면접에서 자주 나오는 네트워크·OS·DB·자료구조·보안·시스템 설계 질문을 실무 상황과 연결해 답변하는 심화 노트.

## 카드 요약

- **네트워크**: HTTP 흐름, TCP/UDP, DNS, TLS, 캐시, 지연 시간 분석.
- **OS**: 프로세스/스레드, 메모리, 동시성, 데드락, 스케줄링.
- **DB**: 인덱스, 트랜잭션, 격리 수준, 락, 정규화/비정규화.
- **자료구조/알고리즘**: 시간복잡도, 해시, 큐, 힙, 그래프를 실무 문제에 적용.
- **보안/운영**: 인증/인가, XSS/CSRF, 로그, 모니터링, 장애 대응.

## 답변 프레임

실무 면접에서 CS 질문을 받으면 아래 순서로 답하면 안정적이다.

1. **정의**: 개념을 한 문장으로 말한다.
2. **동작 흐름**: 내부에서 어떤 순서로 일이 일어나는지 설명한다.
3. **트레이드오프**: 장점, 비용, 실패 상황을 같이 말한다.
4. **실무 연결**: 성능, 장애, 보안, 유지보수 중 어디에 영향을 주는지 붙인다.
5. **경험/예시**: 직접 겪은 사례나 설계 예시로 마무리한다.

## 네트워크

### 브라우저에서 URL을 입력하면?

답변은 보통 아래 흐름으로 전개한다.

1. 브라우저가 URL을 파싱하고 캐시/HSTS를 확인한다.
2. DNS로 도메인을 IP로 변환한다.
3. TCP 연결을 맺고, HTTPS라면 TLS 핸드셰이크를 수행한다.
4. HTTP 요청을 보내고 서버는 응답을 반환한다.
5. 브라우저는 HTML/CSS/JS를 파싱해 렌더링한다.

### 실무 포인트

- 느린 페이지를 볼 때 **DNS, TCP, TLS, TTFB, 다운로드, 렌더링** 중 어디가 병목인지 나눠 본다.
- HTTP/2는 멀티플렉싱으로 여러 요청을 한 연결에서 처리하지만, 서버/프록시/브라우저 지원을 같이 봐야 한다.
- 캐시는 `Cache-Control`, `ETag`, `Last-Modified`, CDN 정책이 함께 동작한다.

### 대표 질문

- “HTTP와 HTTPS는 어떻게 다른가요?”
- “TCP와 UDP는 언제 각각 쓰나요?”
- “DNS 장애가 나면 서비스에는 어떤 영향이 생기나요?”
- “브라우저에서 요청이 느릴 때 어디부터 확인하나요?”

## OS

### 프로세스와 스레드

- **프로세스**: 독립된 주소 공간을 가진 실행 단위.
- **스레드**: 같은 프로세스 안에서 메모리를 공유하는 실행 흐름.
- 스레드는 가볍지만 공유 자원 때문에 race condition이 생길 수 있다.

### 동시성 답변 포인트

- 공유 자원, 임계 구역, 락, 세마포어, 뮤텍스를 연결한다.
- 데드락은 **상호 배제, 점유 대기, 비선점, 순환 대기**가 동시에 만족될 때 생긴다.
- 해결은 예방, 회피, 탐지/복구, 타임아웃 같은 전략으로 나뉜다.

### 실무 연결

- Node/브라우저는 JS가 단일 스레드처럼 보이지만, 런타임과 OS 레벨에서는 I/O, 워커, 이벤트 루프가 함께 동작한다.
- 서버에서 스레드 풀 고갈, 커넥션 풀 고갈, 락 경합은 실제 장애로 이어질 수 있다.

## DB

### 인덱스

인덱스는 읽기 성능을 높이기 위한 별도 자료구조다. 보통 B-Tree 계열을 사용하고, 조건 검색/정렬/범위 조회에 도움을 준다.

주의할 점:

- 인덱스는 쓰기 비용과 저장 공간을 늘린다.
- 선택도가 낮은 컬럼은 효과가 작을 수 있다.
- 복합 인덱스는 컬럼 순서가 중요하다.
- `LIKE '%keyword'`처럼 앞이 열려 있으면 일반 인덱스를 타기 어렵다.

### 트랜잭션

- **Atomicity**: 전부 성공하거나 전부 실패한다.
- **Consistency**: 제약 조건을 깨지 않는 상태를 유지한다.
- **Isolation**: 동시에 실행되어도 서로 영향을 제한한다.
- **Durability**: 커밋된 데이터는 장애 후에도 유지된다.

### 실무 질문

- “결제 API에서 중복 결제를 어떻게 막을 건가요?”
- “인덱스를 추가했는데도 쿼리가 느린 이유는 무엇일까요?”
- “트랜잭션 격리 수준이 낮으면 어떤 문제가 생기나요?”
- “캐시와 DB 사이의 정합성을 어떻게 관리하나요?”

## 자료구조/알고리즘

### 실무에서 자주 쓰는 연결

- **HashMap**: 빠른 조회, 중복 제거, 카운팅.
- **Queue**: 작업 대기열, BFS, 비동기 처리.
- **Heap/Priority Queue**: 상위 N개, 스케줄링, 우선순위 작업.
- **Graph**: 추천, 의존성, 네트워크, 라우팅.
- **Trie**: 자동완성, prefix 검색.

### 답변 포인트

- Big-O만 말하지 말고 입력 규모, 메모리, 정렬 여부, 업데이트 빈도를 같이 본다.
- 평균 복잡도와 최악 복잡도를 구분한다.
- 실무에서는 알고리즘보다 데이터 모델링/캐싱/인덱스가 더 큰 성능 개선일 때도 많다.

## 보안

### 인증/인가

- 인증은 “누구인가”, 인가는 “무엇을 할 수 있는가”다.
- 프론트에서 버튼을 숨기는 것은 UX일 뿐 보안이 아니다. 서버에서 권한을 검증해야 한다.
- 토큰 저장 위치는 XSS/CSRF 위협 모델과 함께 판단한다.

### 웹 취약점

- **XSS**: 신뢰할 수 없는 스크립트가 실행되는 문제. 이스케이프, CSP, 입력 검증으로 줄인다.
- **CSRF**: 사용자의 인증 상태를 악용해 원치 않는 요청을 보내는 문제. SameSite, CSRF 토큰으로 줄인다.
- **SQL Injection**: 쿼리 문자열 조작. prepared statement, ORM 파라미터 바인딩이 기본이다.

## 시스템 설계

### 실무 면접에서 보는 기준

- 요구사항을 먼저 나눈다: 기능 요구사항, 비기능 요구사항.
- 읽기/쓰기 비율, 트래픽, 데이터 크기, 지연 허용치를 가정한다.
- 단일 서버 → LB → 캐시 → DB replica → 큐 → 샤딩 순서로 확장한다.
- 장애 지점을 찾고 fallback/timeout/retry/circuit breaker를 붙인다.

## 좋은 답변 예시 구조

질문: “캐시를 쓰면 어떤 문제가 생길 수 있나요?”

답변 흐름:

1. 캐시는 DB 부하와 응답 시간을 줄이기 위해 사용한다.
2. 하지만 원본 데이터와 캐시 사이에 정합성 문제가 생긴다.
3. TTL, write-through, write-around, cache-aside 같은 전략을 선택할 수 있다.
4. 트래픽이 몰릴 때 cache stampede가 생길 수 있으니 lock, jitter, stale-while-revalidate 등을 고려한다.
5. 실무에서는 데이터 성격에 따라 강한 정합성이 필요한지, 약간 오래된 값을 허용할 수 있는지 먼저 정한다.

## 면접 포인트

1. **CS는 정의보다 적용이다**: “그래서 어떤 장애/성능/보안 문제를 막는가”까지 말한다.
2. **모르면 범위를 좁힌다**: “제가 아는 범위에서는…”으로 시작해 흐름을 설명하고, 모르는 세부는 인정한다.
3. **트레이드오프를 말하면 깊어 보인다**: 성능을 얻으면 복잡도/비용/정합성 문제가 생길 수 있다.
4. **운영 관점을 붙인다**: 로그, 메트릭, 알람, 롤백, 장애 대응까지 언급하면 실무형 답변이 된다.
